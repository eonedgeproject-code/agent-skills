# Database Schema Migrations

## Overview

A migration is a **versioned, reviewed, tested** change to a deployed database's
schema or data. Migrations are code: they live in the repo, run in order, and are
never edited once applied anywhere. The default posture is **zero-downtime and
reversible** — a schema change should not require stopping the service, and you
should be able to roll it back. Editing a shipped migration, or a `DROP` that runs
in the same deploy that stops reading a column, is how you lose data.

**If a schema change isn't a migration in the repo, it doesn't exist** — never
hand-edit a production schema.

## When to Use

- Adding/altering/removing a table, column, index, constraint, enum, or default.
- Backfilling or transforming existing rows.
- Any schema change to a database that is already deployed (staging or prod).
- Introducing a new table for a feature (pairs with `spec-driven-development`).

**When NOT to use:**

- **On-chain account layouts / program state** → `deprecation-and-migration`
  (Anchor account realloc, versioned accounts, data migration instructions).
- Pure application code with no schema impact.
- The very first schema of a not-yet-deployed database (just write the initial
  migration; there is nothing to preserve).

Backend commands and DB conventions defer to `fullstack-standard` →
`references/backend-standards.md`; the pre-ship gate to
`references/definition-of-done.md`.

## Core Principles

1. **Migrations are immutable once applied.** Never edit a migration that has run
   on any shared environment. Fix-forward with a new migration.
2. **One logical change per migration**, one migration set per PR. Reviewable and
   revertible in isolation.
3. **Forward and rollback.** Write the `down`/rollback path unless it is provably
   impossible (a destructive drop) — and say so explicitly when it is.
4. **Separate schema change from data backfill** for large tables. A slow
   `UPDATE` inside a schema migration holds locks and blocks deploys.
5. **Expand then contract.** Never rename or drop in one step against a running
   app. Add the new shape, migrate reads/writes, *then* remove the old shape in a
   later deploy.
6. **Mind locks.** DDL can take heavy locks. On Postgres use
   `CREATE INDEX CONCURRENTLY`, add nullable columns (cheap) before backfilling,
   and avoid rewriting large tables in one statement.
7. **Destructive = explicit approval.** `DROP TABLE/COLUMN`, `TRUNCATE`, type
   narrowing, and non-null-without-default on a big table are high-risk — follow
   `doubt-driven-development` and get sign-off. Back up first.

## The expand–contract pattern (zero-downtime rename)

Renaming `users.email` → `users.email_address` without downtime is **three
deploys**, not one:

```
Deploy 1 — EXPAND
  • migration: ADD COLUMN email_address (nullable), backfill in batches
  • app: dual-write both columns, read old
Deploy 2 — MIGRATE READS
  • app: read email_address, still dual-write
  • migration: backfill any stragglers; add NOT NULL once fully populated
Deploy 3 — CONTRACT
  • app: stop referencing email
  • migration: DROP COLUMN email   (only after nothing reads it)
```

The same shape covers splitting a column, changing a type, or moving data between
tables. The rule: **the schema is always compatible with both the currently
running app and the one you're deploying.**

## Batched backfills

Never backfill a large table in a single statement — it locks rows and bloats the
transaction. Chunk it:

```sql
-- Postgres: backfill in bounded batches, committing between chunks.
UPDATE users
SET email_address = email
WHERE id IN (
  SELECT id FROM users
  WHERE email_address IS NULL
  ORDER BY id
  LIMIT 5000
);
-- Loop from the app/migration runner until 0 rows affected.
```

For heavy backfills, prefer a separate one-off job over the migration runner so a
slow backfill can't block or time out a deploy.

## Per-stack tooling

### Python — Alembic (SQLAlchemy)

```bash
alembic revision -m "add users.email_address"     # or --autogenerate
alembic upgrade head                                # apply
alembic downgrade -1                                # roll back one
alembic history --verbose                           # audit the chain
```

- Review autogenerated migrations by hand — autogen misses renames, enums, and
  server defaults, and may emit a destructive diff.
- Keep `upgrade()` and `downgrade()` both meaningful. Type hints and a docstring
  per migration (house Python style, `ruff`, line length 100).

### TypeScript — Prisma / Drizzle

```bash
# Prisma
prisma migrate dev --name add_email_address   # dev: generate + apply
prisma migrate deploy                           # prod: apply committed migrations
prisma migrate status                           # drift/pending check

# Drizzle
drizzle-kit generate   # emit SQL from schema
drizzle-kit migrate    # apply
```

- Commit the generated SQL; never run `db push` against a shared database (it
  mutates schema without a migration record).
- `strict: true`; the schema is the typed contract the app imports — no `any`.

### Rust — sqlx

```bash
sqlx migrate add add_email_address     # creates timestamped up/down .sql
sqlx migrate run                        # apply
sqlx migrate revert                     # roll back one
```

- Access the DB directly with `sqlx` (no repository layer — `fullstack-standard`
  minimal abstraction). `sqlx migrate build-script` keeps compile-time query
  checks honest.

## Testing & CI

- **Round-trip in CI:** apply `up` then `down` then `up` on a scratch database and
  assert it succeeds — a rollback that doesn't run is not a rollback.
- Test migrations against **prod-like data volume** for anything with a backfill or
  an index build; a migration that's instant on 10 rows can lock a table for
  minutes on 10M.
- Add/adjust tests for code whose behavior depends on the new schema
  (`test-driven-development`).
- Gate on the Definition of Done (`fullstack-standard` →
  `references/definition-of-done.md`) before shipping.

## Safety checklist before a production migration

- [ ] Reversible, or the irreversibility is called out and approved.
- [ ] No destructive step in the same deploy that stops using the old shape.
- [ ] Backfills are batched; index builds are `CONCURRENTLY` (Postgres).
- [ ] Backup/snapshot taken before any destructive change.
- [ ] Rollback path tested (`down` actually runs).
- [ ] Reviewed as its own PR; lockstep with the app deploy that needs it.
